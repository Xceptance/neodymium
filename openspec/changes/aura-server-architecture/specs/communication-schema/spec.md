## ADDED Requirements

### Requirement: Structured Logging Protocol (JSON Lines)
To guarantee high-performance, non-blocking, and crash-resilient data capture, the Neodymium runner SHALL emit all test suite lifecycle milestones, interactive steps, and diagnostic captures as a chronological stream of JSON Lines (JSONL) to a dedicated SLF4J logging channel named `aura.events`.

---

### Requirement: Run-Scoped Directory & Process Isolation
To prevent concurrent write conflicts, file-locking bottlenecks, and data corruption when running tests concurrently across multiple threads or JVM forks, each test execution session MUST write its events to a uniquely named log file within a dedicated, run-scoped directory.

#### Scenario: Verify isolated log file creation
- **WHEN** a test run starts
- **THEN** Neodymium creates a dedicated directory: `target/aura-results/run_<runId>/`
  - Where `runId` is a unique timestamp-based identifier (e.g. `20260527_210532_uuid`).
- **AND** the logging framework writes events to a fork-specific log file: `aura-events-<forkId>.jsonl`
  - Where `forkId` is a unique process, fork, or worker index (e.g. `fork_1`, `fork_2`).

---

### Requirement: Multi-File Ingestion & Consolidation
The Aura system (or Neodymium's offline generator) SHALL support scanning the run-scoped directory, concurrently parsing all `aura-events-*.jsonl` files generated by different execution processes, and consolidating them into a single coherent test run state.

#### Scenario: Ingest and consolidate concurrent log files
- **WHEN** the test run completes and the `run_<runId>` folder is parsed
- **THEN** the compiler reads all `aura-events-*.jsonl` files in that folder.
- **AND** it consolidates their execution events to write:
  - `run-info.json`: Standardized aggregated summary of results across all forks.
  - `test-cases/<className>_<methodName>.json`: Chronological step traces for each test case.


---

### Requirement: Suite and Test Lifecycle Events
The logging protocol MUST support lifecycle events to record execution boundaries, environment information, and overall run metrics.

#### Scenario: Log SUITE_START and SUITE_FINISH events
- **WHEN** the test suite run starts
- **THEN** Neodymium logs a `SUITE_START` event containing:
  - `runId` (unique timestamp-based identifier)
  - `gitInfo` (branch, commit SHA)
  - `environment` (browser, platform, screenResolution)
- **WHEN** the test suite run completes
- **THEN** Neodymium logs a `SUITE_FINISH` event containing:
  - `runId`
  - `summary` (totalTests, passedCount, failedCount, warningCount, skippedCount, durationMs)

---

### Requirement: Step Trace Event Schema
Each interactive browser step intercepted by Neodymium's runner MUST be logged as a discrete `STEP_EXECUTION` event within the test case trace.

#### Scenario: Log STEP_EXECUTION step properties
- **WHEN** a test step completes execution
- **THEN** Neodymium logs a `STEP_EXECUTION` event containing:
  - `index` (1-indexed chronological position)
  - `instruction` (fully stripped natural language instruction)
  - `stepType` (one of `STANDARD`, `LAYOUT`, `VISUAL`, `USABILITY`, `LANGUAGE`)
  - `status` (one of `PASSED`, `FAILED`, `WARNING`, `SKIPPED`)
  - `durationMs` (execution time in milliseconds)
  - `actions` (array of physical WebDriver/Selenide action records)
  - `screenshotPath` (relative path to the screenshot viewport file, if captured)
  - `domSnapshotPath` (relative path to the outerHTML DOM snapshot file, if captured)

---

### Requirement: Observational Visual Anomalies Log
The logging protocol MUST support embedding visual and layout linter audit results (like `Aura Glance` observations) directly inside the step execution payload.

#### Scenario: Log step with visualAnomalies list
- **WHEN** an observational check or usability audit detects visual discrepancies
- **THEN** Neodymium appends a `visualAnomalies` array inside the logged `STEP_EXECUTION` event, where each anomaly contains:
  - `type` (e.g., `OVERLAP`, `CLIPPING`, `CONTRAST`, `VISUAL_DRIFT`, `DEAD_END`)
  - `severity` (e.g., `ERROR`, `WARNING`)
  - `description` (the AI's textual observation of the bug)
  - `selector` (the CSS selector or locator, if resolved)
  - `coordinates` (bounding box coordinates: `{ x, y, width, height }`)

---

### Requirement: Non-Blocking Soft Step Warning Log
When a test step tagged with `(soft)` fails or detects non-critical visual anomalies, the runner SHALL log the step with `status: "WARNING"` and continue execution without halting the test.

#### Scenario: Verify soft step warning logging
- **WHEN** a playbook step `Observe visual consistency (soft) (visual)` fails with visual anomalies
- **THEN** Neodymium:
  - Emits a `STEP_EXECUTION` event with `status: "WARNING"` and populates the `visualAnomalies` array.
  - Logs a warning message to the standard developer console: `⚠️ Soft step warning: [anomaly descriptions]`.
  - Advances to the next step without raising a test-halting exception.

---

### Requirement: Optional Step Silent Bypassing
When a test step tagged with `(optional)` fails due to missing elements or action failures, the runner SHALL log the step as either `"SKIPPED"` or `"PASSED"` (if bypassed by design) and continue execution without raising a warning.

#### Scenario: Verify optional step bypassing
- **WHEN** a playbook step `Click "Popup Close" (optional)` fails because the popup button is not present
- **THEN** Neodymium:
  - Emits a `STEP_EXECUTION` event with `status: "SKIPPED"` (or `"PASSED"` if bypassed by design).
  - Logs a low-priority debug message to the developer console: `Optional step skipped: element not found`.
  - Continues running subsequent steps with zero warning notifications.

---

### Requirement: Server-Independent Execution (Offline Resiliency)
The test execution runner (Neodymium) MUST operate and complete all test scenarios with zero performance degradation or runtime exceptions if the Aura Server is offline, unreachable, or crashes mid-run.

#### Scenario: Verify offline runner execution
- **GIVEN** the Aura Server is offline or crashes during a test run
- **WHEN** Neodymium executes tests
- **THEN** the runner continues to execute all steps to completion without raising socket, network, or server-connectivity exceptions.
- **AND** Neodymium successfully records all lifecycle and step events to the local `aura-events.jsonl` file.

---

### Requirement: Post-Crash Ingestion and Recovery
The Aura Server (upon startup or manual re-index trigger) SHALL be capable of parsing partially or fully completed local `aura-events.jsonl` files retrospectively to rebuild its relational database index with zero data loss.

#### Scenario: Recover test data after server crash
- **GIVEN** the Aura Server crashed during an active test run
- **WHEN** the Aura Server restarts
- **THEN** the server's directory scanner detects the `aura-events.jsonl` log file.
- **AND** the server reads the file sequentially from the beginning (or from its last saved line offset) to fully reconstruct and index the completed execution steps and metrics.
